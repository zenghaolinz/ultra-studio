# Ultra Studio 0.7.0 Generation Task Center Design

## Goal

Deliver one coherent 0.7.0 vertical slice: generation jobs remain visible and controllable while chat continues, update without chat-side polling, and can be cancelled or retried individually after failures or application restarts.

## Scope

0.7.0 includes task status events, single-task cancellation, retry from persisted inputs, restart recovery, a shared frontend task store, a unified task center for image/video/3D jobs, and concurrency/lifecycle tests.

0.7.0 excludes VRAM-aware scheduling, broad settings refactoring, a full MCP protocol upgrade, and unrelated chat orchestration refactoring.

## Architecture

`generation_tasks` remains the source of truth. All task mutations pass through `sidecar/services/generation_tasks.py`, which publishes a normalized task snapshot after a committed state change. A dedicated event broker exposes those snapshots as a Sidecar SSE stream. The Tauri bridge subscribes once and emits `generation-task-updated` events to the webview.

The frontend owns one Zustand task store. The queue center, generation history, and chat cards consume that store. They may perform one initial snapshot fetch and reconnect recovery fetch, but must not maintain independent polling loops.

Retry creates a new task linked to the original task rather than mutating terminal history. Persisted task inputs include enough structured request data for the task dispatcher to reconstruct supported image, video, and 3D work. Unsupported legacy tasks receive an explicit non-retryable response.

## Backend Components

- `generation_tasks.py`: task CRUD, legal state transitions, single-task cancel, retry record creation, serialization, and event publication after commit.
- `generation_events.py`: bounded in-process subscriber queues and SSE heartbeat support. Slow subscribers receive the newest state through snapshot resynchronization rather than blocking workers.
- `generation_dispatcher.py`: maps persisted task type/request data to existing generation entry points. It contains no chat behavior.
- `routes/generation_tasks.py`: snapshot, event stream, cancel, and retry HTTP endpoints.
- `main.py`: startup recovery and router registration.

## Task State Model

Statuses remain `queued`, `running`, `success`, `error`, and `cancelled`. Legal transitions are:

- `queued -> running | cancelled | error`
- `running -> success | error | cancelled`
- terminal states never transition
- retry creates a new `queued` task with `retry_of_task_id`

On restart, stale `running` tasks become `error` with a restart-specific error code and remain retryable when structured request data exists. Queued tasks remain queued only if the dispatcher can resume them; otherwise startup marks them as interrupted errors to avoid ghost jobs.

## Cancellation

Cancellation targets one task ID. Queued tasks are atomically cancelled. Running tasks are marked with cancellation intent and the dispatcher invokes the relevant runtime interrupt when supported. A late worker result cannot overwrite a cancelled row.

The existing global 3D cancel command remains as a compatibility wrapper but the new UI uses the task-specific endpoint.

## Frontend

`generationTaskStore.ts` performs initial hydration, applies task update events, reconnects after Sidecar restarts, and exposes selectors/actions. `GenerationTaskCenter` provides status filters, current queue ordering, cancel/retry actions, errors, and output restoration/reveal actions.

Chat generation cards subscribe by task ID. Removing their 2.5-second polling loop is an explicit acceptance criterion.

## Chat Route Guardrail

`sidecar/routes/chat.py` must not gain scheduling, persistence, retry, cancellation, or event-broadcast logic. Its final line count must not exceed the pre-0.7.0 baseline of 1,940 lines. Any necessary wiring must be delegated to generation services and offset by extraction when it would otherwise grow the file.

## Error Handling

- Missing task: HTTP 404.
- Illegal transition or retry of an active task: HTTP 409.
- Missing retry payload or unsupported legacy task: HTTP 422 with a stable error code.
- Event disconnect: frontend reconnects with bounded backoff and performs snapshot reconciliation.
- Subscriber overflow: broker drops stale queued notifications and sends a resync marker.

## Testing

Unit tests cover migrations, transition guards, single-task cancellation, retry linkage, serialization, and broker isolation. Route tests cover snapshot/SSE/control errors. Frontend type checks cover store/component contracts. Integration tests cover concurrent chat and generation, restart recovery, late completion after cancellation, and two simultaneous event subscribers.

## Acceptance Criteria

- Chat remains usable while generation runs.
- Task updates appear without chat-side intervals.
- A user can cancel one task without cancelling unrelated work.
- A failed or interrupted supported task can be retried as a new task.
- Restart never leaves a worker-less task displayed as running.
- Image, video, and 3D tasks appear in one task center.
- `chat.py` remains at or below 1,940 lines and contains no new queue business logic.
